Product Requirements

HADR Monitor

An unattended agent that watches three disaster feeds and publishes one trustworthy morning situation report.

Shape B · 9/9 fit Stack: Python · uv · GitHub Actions Source: docs/PRD.md Dated 2026-07-08

01 Goal

Every morning at 08:30 Singapore time, publish a single self-contained dashboard carrying every reportable disaster event — deduplicated across GDACS, USGS and ReliefWeb, filtered to what actually matters, with what happened, where, how bad, and who is affected. It runs unattended, stays honest when a feed is down, corrects itself when events change, and is quiet on quiet mornings — without a language model ever deciding whether to wake up.

The problem it removes: staying aware today means watching three noisy, differently-shaped, SLA-free feeds that double-report the same event, revise or retract it after publication, and emit far more noise than signal. There is no low-effort way to get a once-a-day picture that distinguishes a calm world from a dead feed.

02 Three vertical slices

Each slice ends in a demo-able dashboard — no horizontal “build all the models first” layers. The renderer ships in slice 1 so every later slice proves itself by changing the published page. (The full eight-step breakdown lives in SLICES.md; these are its three headline increments.)

SLICE 1 · THE SKELETON

Live earthquakes, deduplicated, on the page

Fetch USGS and GDACS, reconcile their records into canonical events with our own IDs (dedup via the live-verified GDACS sourceid → USGS alias join), persist to data/state.json, and render the self-contained dashboard.

Demo: the same quake reported by USGS and GDACS shows as one card with both source links — and a revised magnitude updates that card instead of adding a second.

SLICE 2 · THE DECISION

Report only what matters, and record what changed

Apply the impact-based threshold — GDACS Orange/Red, USGS PAGER yellow+, an escalation of a tracked event, or a human-curated ReliefWeb entry. Everything below the bar is tracked in state but hidden. Each run produces a dated edition with a changelog of escalations, downgrades and retractions; a run with nothing reportable renders a one-line quiet edition.

Green M4.6 → tracked, hidden Orange flood → reported Green→Red → leads changelog
Demo: escalate a tracked event between runs → it leads the next changelog; an empty morning → the deterministic quiet edition, no model call.

SLICE 3 · TRUSTWORTHY & UNATTENDED

Assessment, coverage honesty, and a schedule

Add model-written assessments for reportable events (skipped entirely on quiet mornings), explicit coverage warnings when a feed is stale or down, and the hourly + 08:30 SGT schedule — plus an intraday flash re-render when a new Red-level event appears between editions.

Demo: it runs itself on the schedule and commits the result; a dead feed shows “GDACS unreachable since 03:10 SGT”; a new Red event flashes to the dashboard within the hour, not the next morning.

03 Data model

One schema-versioned JSON file, committed to the repo by the workflow (never hand-edited). Its git history is the audit log of what the agent believed and when. A canonical event merges all source records for one real-world occurrence and keeps every source alias so matching survives preferred-ID flips. GDACS alert level and USGS PAGER level are kept as separate fields — they are different impact models and are never merged. 

// data/state.json
{
  "version": 1,

  "events": {                        // keyed by our own canonical id
    "evt-2026-0714": {
      "hazard": "EQ",               // EQ | TC | FL | VO | DR | WF
      "name": "M5.0 — 15 km NNE of Xunchang, China",
      "aliases": [                  // every identifier any feed used
        "usgs:us6000taui", "gdacs:1550709", "glide:EQ-2026-000093-CHN"
      ],
      "location": { "lat": 28.5871, "lon": 104.7549, "place": "China" },
      "origin_time": "2026-07-08T02:08:48Z",   // UTC; SGT only at render
      "magnitude": { "value": 5.0, "type": "mw", "depth_km": 10 },  // EQ only
      "gdacs_alert": "Green",       // Green|Orange|Red|null — impact model A
      "pager_alert": null,          // green|yellow|orange|red|null — impact model B
      "reported": false,            // has it ever cleared the threshold?
      "flash_published": false,     // Red flash fired this spell; cleared on drop below Red
      "status": "active",           // active | aged_out | retracted
      "first_seen": "2026-07-08T02:49:03Z",
      "last_changed": "2026-07-08T02:49:03Z",
      "source_refs": {
        "usgs": "https://earthquake.usgs.gov/.../us6000taui",
        "gdacs": "https://www.gdacs.org/report.aspx?eventid=1550709"
      }
    }
  },

  "feed_status": {                   // per feed, tracked independently
    "gdacs":     { "last_success": "...", "feed_generated_at": "...", "consecutive_failures": 0 },
    "usgs":      { "last_success": "...", "feed_generated_at": "...", "consecutive_failures": 0 },
    "reliefweb": { "last_success": "...", "feed_generated_at": "...", "consecutive_failures": 0 }
  },

  "edition_marker": {                // what the reader has already been told
    "last_edition_at": "2026-07-07T00:30:00Z",
    "acknowledged_changes": ["evt-2026-0712:escalation", "..."]
  }
}

Join order for merging records into one event: shared GLIDE number → GDACS detail sourceid matched into the USGS alias set → heuristic (same hazard, origin time within ±30 min, epicentres ≤ 250 km). Absence from a feed’s rolling window is aged out, never a retraction — only an explicit deletion signal retracts.

04 Out of scope

Deliberate exclusions — as much a decision as what’s in.

  • Hazards without a source in the stack — conflict, displacement, epidemics, tsunami warnings, landslides, heatwaves. Drought is in only as far as GDACS itself raises it.
  • Feeds beyond GDACS, USGS and ReliefWeb — no ACLED, WHO, NOAA/PTWC, FIRMS, etc., however useful.
  • Notification channels beyond the dashboard — no email, Slack or outbound RSS. The committed dashboard.html is the only surface.
  • Intraday alerting below Red level — sub-daily publishing fires only for new Red events; everything else waits for the 08:30 edition.
  • Historical backfill / archive queries — state tracks the active window, not a research archive.
  • Region-specific weighting — global scope; severity does the filtering, not geography.
  • Document-volume as a severity signal — report count tracks international attention, not impact.
